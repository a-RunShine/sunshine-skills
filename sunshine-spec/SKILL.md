## 共享机制：文档生成 → 审查 → 优化

每份文档（spec / plan）生成后必须经过 **Explore 子 agent** 的结构化审查才能进入用户审批。LLM 自我提示式的自检只能算基线，真正的质量把关交给独立的审查 agent。

### 流程

```
初稿 → 维度挑选 → Explore 审查（1 次）→ 报告解析
                                        │
                  ┌─────────────────────┼─────────────────────┐
                  ▼                     ▼                     ▼
           critical/major=0       有可消化的问题         重写后仍有未消化关键问题
                  ↓                     ↓                     ↓
           进入审批 / 自动推进    skill 一次性重写              升级用户
```

### 通用规则

- 审查 agent 只读，绝不修改任何文件
- **每次对话前，skill 必须用 AskUserQuestion 询问审查策略**（单轮 / 多轮），一次询问，整个任务都遵循（用户选择单轮，整个工作都单轮审查）

- 改动点用 HTML 注释 `<!-- review-fix: {ISSUE-id} -->` 标记
- **重写后不再调用 Explore agent 复审**——重写后直接根据「未消化问题」评估：critical=0 且 major<3 即通过，否则升级用户
- 升级用户条件见「迭代优化机制」小节
- 两份维度池见各阶段「审查维度池」小节
- Explore agent prompt 模板与动态挑选规则见下文共享小节

### 维度动态挑选规则

skill 在调用审查 agent 前采集四类信号：

```
S = {
  doc_type:  'spec' | 'plan',
  task_size: 'xs' | 's' | 'm' | 'l' | 'xl',   // 从任务描述长度/复杂度估算
  domain:    'cli' | 'web' | 'lib' | 'infra' | 'data' | 'ui' | 'other',
  features:  { i18n, auth, perf_critical, network, file_io, ... }
}
```

按以下步骤筛选维度（≤ 8 个，给 agent 保留注意力）：

1. **必选维度全选**（每个维度池中"必选"列）
2. **按适用条件追加**：
   - `task_size ≥ m` → 启用 YAGNI（S7）/ 抽象粒度（P8）
   - `features ⊇ {i18n}` → 启用跨文化中立（S10）
   - `features ⊇ {perf_critical, network, file_io}` → 启用非功能量化（S9）
   - `domain ∈ {cli, lib, infra}` → 启用语言对齐（P7）
3. **截断至 8 个**：critical 全部 > major 按相关性 > minor
4. **输出维度选择声明**给 agent：「本轮选中维度：S1, S2, S3, …（共 N 个）」

每个被选中的维度，skill 生成一段**判定细化文本**注入到 agent prompt（细化来源 = 维度表「判定标准」+ 上下文拼接，不引入新的 LLM 决策）。

### 迭代优化机制（按用户策略执行）

#### 第一步：询问审查策略

每份文档（spec / plan）**生成前**，skill 必须用 `AskUserQuestion` 询问审查策略：

```
question: "这次的审查策略？"
options:
  - 单轮（推荐）：写好 → 审查 → 自动重写 → 评估 → 结束
  - 多轮（最多 3 轮）：写好 → 审查 → 重写 → 审查 → 重写 → …… → 通过或达 3 轮
header: "审查策略"
```

**默认行为**：用户不指定 / 跳过时，**默认单轮**（节省 token）。

#### 第二步：按策略执行

**单轮**（写好 → 审查 → 重写 → 评估）：

```
report = ExploreAgent.review(doc)        # 审查 1 次
critical_open, major_open = 统计(report)

if critical_open == 0 and major_open == 0:
  → 进入审批 / 自动推进    # 审查一次就够
else:
  doc = Skill.rewrite(doc, report)       # 重写 1 次（不再次审查）
  remaining = 评估(rewrite 后的未消化问题)
  if remaining.critical > 0 or remaining.major >= 3:
    → 升级用户
  else:
    → 进入审批 / 自动推进（minor 可接受）
```

**多轮**（最多 3 轮，每轮：审查 → 重写 → 评估）：

```
round = 1
max_rounds = 3
loop:
  report = ExploreAgent.review(doc)
  critical_open, major_open = 统计(report)

  if critical_open == 0 and major_open == 0:
    BREAK → 进入审批 / 自动推进    # 通过

  if round == max_rounds:
    ESCALATE → 升级用户           # 达上限仍未通过
    BREAK

  doc = Skill.rewrite(doc, report)        # 重写（不立即审查）
  round += 1
```

**关键规则**：

- **重写后不立即再次审查**——避免反复 LLM 调用；下一轮审查在下次循环顶部进行
- 单轮 = 1 次审查 + 最多 1 次重写
- 多轮 = 最多 3 次审查 + 最多 3 次重写（每轮末尾才重写）

**重写规则**：

- 合并同 dimension 多条问题为 1 个改写目标
- 优先级：critical > major > minor；同类按 dimension 排序
- 每次重写最多处理 5 处；超出说明规划期就崩了，应升级用户
- 保留文档原有结构与编号体系
- 在改动位置加 HTML 注释 `<!-- review-fix: {ISSUE-id} -->`
- 重写结束在审批模板前加一行声明："基于 Explore agent 报告（round {N}，N 个 issue）已重写 M 处"

**升级用户的触发条件（任一满足）**：

| 条件                            | 升级内容                         |
| ----------------------------- | ---------------------------- |
| 单轮：重写后仍有 ≥ 1 个 critical 未消化   | 列出未消化的 critical + 重写记录，请用户裁决 |
| 单轮：重写后仍有 ≥ 3 个 major 未消化      | 同上，列 major                   |
| 多轮：3 轮后仍有 ≥ 1 个 critical 未消化  | 列出全部未消化问题 + 3 轮处理记录          |
| 多轮：3 轮后仍有 ≥ 3 个 major 未消化     | 同上                           |
| 单轮 issue 总数 > 15              | 文档质量崩溃，提示重写而非修补              |
| critical 中出现结构性（模板不适配、维度无法满足） | 暂停自动重写，请用户确认是否调整 skill 本身    |

**升级模板**：

```
⚠️ 文档质量升级：
第 {doc_type} 文档（{策略：单轮/多轮}）经 {轮数} 轮后仍有 N 个未消化问题：
- [critical] ISSUE-id：problem（位置：location）
- ...
已尝试的修改：（概述）
请选择：① 接受当前版本继续 ② 手动修改后重新审查 ③ 回到前一阶段重新生成
```

# Spec 驱动开发

把想法变成可运行的代码，中间经过两份递进文档：

```
spec.md（做什么）→ plan.md（怎么做）
```

每份文档在前一份基础上细化。

<HARD-GATE>
spec 和 plan 全部生成并获得用户批准之前，禁止编写任何实现代码。无论项目看起来多简单，一律走完流程。
</HARD-GATE>

## 反模式：「这个太简单了，不需要写 spec」

每个项目都要走这套流程。一个工具函数、一次配置改动、一个单文件模块，全都要。越是「简单」的项目，未被审视的假设越多，返工的概率越高。文档可以写得短，但必须存在、必须被审批。

## 两份文档的职责

| 文档      | 回答什么 | 包含什么                                            |
| ------- | ---- | ----------------------------------------------- |
| spec.md | 做什么  | 背景、目标、功能需求、非功能需求、边界（明确哪些留给后续）、验收标准          |
| plan.md | 怎么做  | 架构概览、组件划分、核心接口与数据结构、模块交互、技术决策、文件组织（按目录与文件级别） |

## 流程总览

```
用户想法
    │
    ▼
阶段一：需求澄清
    │
    ├─→ 问审查策略（单轮 / 多轮）──→ spec.md ──→ Explore 审查 + 自动重写（按策略）──→ 用户审批（强）★
    │                                                                              │ 不通过：修改
    │                                                                              ▼ 通过
阶段二：技术设计
    │
    ├─→ 问审查策略（单轮 / 多轮）──→ plan.md ──→ Explore 审查 + 自动重写（按策略）──→ 软提示（默认推进）
    │                                                                              │ 用户主动暂停/修改
    │                                                                              ▼ 默认推进
阶段三：开发（按 plan.md 执行）
    │
    ▼
阶段四：验收（按 spec.md 验收标准检查）
```

**审查策略**（每文档生成前询问，默认单轮）：

| 策略         | 流程                                     | 适用        |
| ---------- | -------------------------------------- | --------- |
| 单轮（推荐）     | 写好 → 审查 → 自动重写 → 评估 → 结束               | 常规任务、需求清晰 |
| 多轮（最多 3 轮） | 写好 → 审查 → 重写 → 审查 → 重写 → …… → 通过或达 3 轮 | 关键决策、复杂模块 |

**审批强度对照**：

| 阶段      | 审批强度        | 触发暂停的关键词                       |
| ------- | ----------- | ------------------------------ |
| spec.md | ★★★ 强（阻塞）   | 默认即阻塞等待用户确认                    |
| plan.md | ★★☆ 软（异步提示） | 「暂停 plan」/「改 plan」/「重做 plan」 |

## 阶段一：需求澄清 → spec.md

**输入：** 用户的初步想法或粗略描述
**输出：** spec.md

### 步骤 1：了解上下文

阅读现有代码、文档和最近的提交记录，搞清楚当前状态和本次要做的新东西。

### 步骤 2：澄清需求

**一次只问一个问题。** 能用选择题就不用开放题。

关注点：

- 目的和动机——为什么要做这个？
- 成功标准——怎么判断做完了？
- 边界——哪些明确不做？
- 约束——性能、兼容性、安全性等

如果需求涉及多个独立子系统，立刻指出来，先帮用户拆分成子项目再深入细节。

### 步骤 3：提出方案

提出 2-3 种方案，说清各自的优劣和你的推荐。推荐方案放第一个，解释为什么推荐它。

### 步骤 4：分段呈现 spec

**逐段呈现**，每段确认后再展示下一段：

1. 背景与目标
2. 功能需求（F1, F2, ...）
3. 非功能需求（N1, N2, ...）
4. 不做的事
5. 验收标准

### spec.md 模板

```markdown
# [标题] Spec

## 背景
（要解决什么问题，当前已有什么）

## 目标
- ...

## 功能需求
- F1: ...
- F2: ...

## 非功能需求
- N1: ...

## 不做的事
- ...

## 验收标准
- AC1: ...
- AC2: ...
```

### 写作规则

- **聚焦行为描述。** 「提供一个主入口，接收环境上下文和可选配置，输出完整的 System Prompt 文本」——而非「提供 `BuildSystemPrompt(env, opts)` 函数」
- **保持语言无关。** 同一份 spec 应该适用于 Go、Java 和 Python
- **方法名、类名、数据结构定义留给 plan.md**
- **保持实现无关的抽象层级**，具体文件路径留给 plan.md 的文件组织章节
- **每个小节写完整**，所有内容就绪后再提交审批
- **每条功能需求至少对应一条验收标准**

### 自检（作为 review 输入基线）

> 自检不再独立执行，其内容已并入对应 review 维度。最终判定以 Explore agent 报告为准。

历史自检项保留如下，仅作为 skill 自我提示：

1. **占位符扫描** → S1（review 阶段覆盖）
2. **语言泄漏** → S2（review 阶段覆盖）
3. **歧义检查** → S3（review 阶段覆盖）
4. **范围检查** → S4, S8（review 阶段覆盖）
5. **验收覆盖** → S5, S6（review 阶段覆盖）

### 审查维度池（spec.md）

| ID  | 维度       | 判定标准                                        | 适用条件                | 严重度      |
| --- | -------- | ------------------------------------------- | ------------------- | -------- |
| S1  | 占位符扫描    | 无 TBD/TODO/`<待填>`/空小节                       | 必选                  | critical |
| S2  | 语言泄漏     | 无方法名（驼峰/下划线）、类型签名、import、build/test/lint 命令 | 必选                  | critical |
| S3  | 歧义检测     | 每条 F/N 需求只对应一种解读；含「可能/也许/通常」等弱化词需重写         | 必选                  | major    |
| S4  | 边界覆盖     | 「不做的事」覆盖了用户常见的误解和扩展冲动                       | 必选                  | major    |
| S5  | AC 可观测性  | 每条 AC 可被外部观察（运行 X 看 Y）；无「代码正确」这类内部断言        | 必选                  | critical |
| S6  | F→AC 映射  | 每条 F 至少对应一条 AC；孤儿 AC 不超过 1 条                | 必选                  | major    |
| S7  | YAGNI 纯净 | 不出现 spec 没要求但用户可能喜欢的「扩展点/钩子/预留」             | `task_size ≥ m`     | major    |
| S8  | 单一周期范围   | 估算实现可在单一开发周期（≤ 数天）完成；超出则提示拆项目               | 必选                  | major    |
| S9  | 非功能需求量化  | N 需求给出量化指标或明确的「无要求」，避免「性能好」这种空话             | spec 含 N 章节         | minor    |
| S10 | 跨文化中立    | 避免对中文/英文用户隐含偏向（示例、术语）                       | `features ⊇ {i18n}` | minor    |

### 询问审查策略（spec.md）

在生成 spec.md **之前**，用 `AskUserQuestion` 询问：

```
question: "spec.md 这次的审查策略？"
options:
  - 单轮（推荐）：写好 → 审查 → 自动重写 → 评估 → 结束
  - 多轮（最多 3 轮）：写好 → 审查 → 重写 → 审查 → 重写 → …… → 通过或达 3 轮
header: "审查策略"
```

未指定则默认单轮。

### 审查 agent 调用

按「共享机制 → Explore Agent Prompt 模板」调用 Explore agent：

- `doc_path` = 当前 spec.md 路径
- `doc_type` = `spec`
- `prior_docs` = None（plan 尚未生成，以 spec 自身为校验基准）
- 必选维度：S1, S2, S3, S4, S5, S6, S8
- 按动态挑选规则追加：S7（task_size ≥ m）、S9（spec 含 N 章节）、S10（features ⊇ {i18n}）

收到报告后按「共享机制 → 迭代优化机制」按用户策略处理：

- **单轮**：审查 1 次 → 重写 1 次（最多 5 处，加 `<!-- review-fix: ISSUE-id -->` 标记）→ 评估 → 结束
- **多轮**：审查 → 重写 → 审查 → 重写 → …… → 通过或达 3 轮
- 任何一轮重写后仍有未消化关键问题 → 升级用户

### 用户审批

> spec.md 已生成。请 review：
> 
> - 功能需求是否完整？
> - 有没有遗漏的边界情况？
> - 「不做的事」是否合理？
> - 验收标准是否可观测？
> 
> **审查报告摘要**（Explore agent 第 {round} 轮）
> 
> - 选中维度：{N} 个（{dim_ids}）
> - 发现问题：critical {n_c} / major {n_m} / minor {n_mi}
> - 本轮基于报告已自动重写 {n_fixed} 处，剩余 {n_open} 处
> 
> 确认后进入技术设计阶段。

**等待用户明确批准。** 如果用户要求修改，修改后重新走审查 + 审批。

## 阶段二：技术设计 → plan.md

**输入：** 已批准的 spec.md
**输出：** plan.md

### 流程

1. 重新阅读已批准的 spec.md
2. 设计满足所有功能需求的架构
3. 定义核心数据结构和接口
4. 画出模块间的交互和数据流
5. 记录关键技术决策及其理由
6. **逐段呈现**，每段获得用户确认

### plan.md 模板

```markdown
# [标题] Plan

## 架构概览
（组件/模块划分，每个组件一段话）

## 核心数据结构

### [结构体名]
（字段定义及说明）

### [接口名]
（方法签名及用途）

## 模块设计

### [模块 A]
**职责：** ...
**对外接口：** ...
**依赖：** ...

### [模块 B]
...

## 模块交互
（调用链、数据流。哪个模块调哪个，什么顺序。）

## 文件组织

### 询问审查策略（plan.md）

在生成 plan.md **之前**，用 `AskUserQuestion` 询问：

```

### 审查 agent 调用

按「共享机制 → Explore Agent Prompt 模板」调用 Explore agent：

- `doc_path` = 当前 plan.md 路径
- `doc_type` = `plan`
- `prior_docs` = 已批准的 spec.md
- 必选维度：P1, P2, P3, P4, P5, P6, P9

收到报告后按「共享机制 → 迭代优化机制」按用户策略处理（单轮 / 多轮）。

### 软提示（弱审批）

plan.md 不阻塞用户审批——技术决策可以再调整，但默认自动推进。如果用户主动查看则暂停等待。

**默认推进摘要**（异步展示，不阻塞）：

> plan.md 已生成。
> 
> - 架构划分：[一句话]
> - 核心接口：[列表]
> - 关键技术决策：[列表]
> 
> **审查报告摘要**（Explore agent 第 {round} 轮）
> 
> - 选中维度：{N} 个（{dim_ids}）
> - 发现问题：critical {n_c} / major {n_m} / minor {n_mi}
> - 本轮基于报告已自动重写 {n_fixed} 处，剩余 {n_open} 处
> 
> 默认推进至开发阶段。如需调整 plan，说「暂停 plan」或「改 plan」。

**用户拦截时机**：

- 用户主动说"暂停 plan" / "改 plan" / "重做 plan" → 暂停自动推进，渲染 plan.md 等候修改
- 用户主动说"plan 这块我要看" → 暂停，渲染 plan.md 全文
- 其他情况默认自动推进至开发阶段

## 阶段三：开发

spec 和 plan 全部通过审批，开始实现。

**宣告：** 「spec 和 plan 已全部通过审批，按 plan.md 开始开发。」

### 流程

1. 读 plan.md 的「模块设计」「文件组织」「技术决策」，按 plan.md 的模块划分落地代码
2. 按模块依赖顺序逐个实现：
   - 写完模块的接口与主体逻辑
   - 写对应的单元测试
   - **先有证据再下结论**——先跑命令、看输出，再报状态
   - 编译 + 测试通过后才标记该模块完成
3. 如果被阻塞：停下来问，不要猜
4. 所有模块完成后进入阶段四

### 规则

- 按 plan.md 的设计执行，除非被阻塞否则不自由发挥
- 每完成一个模块必须跑「编译 + 该模块的单元测试」，「应该没问题」不算证据
- 验证不通过就先修，修好再往下走
- 每个模块或每组逻辑相关的模块完成后提交代码

## 阶段四：验收

### 流程

1. 读 spec.md 的「验收标准」（AC1、AC2、...）和「功能需求」
2. 对每条 AC 逐项执行：
   - 跑对应的验证（运行 X，看 Y）
   - 记录实际结果和证据（命令输出、观察到的行为）
   - 标记通过/不通过
3. 如果有不通过的：修复、重新验证、报告
4. 向用户呈现最终报告

### 规则

- **先有证据再下结论。** 先跑命令，看输出，然后再报告。
- 报告**实际结果**，不是预期结果。
- 有不通过的条目不丢人——说明 spec 的 AC 发挥了作用。修好重跑即可。
- 验收粒度按 spec 写明的可观测行为对齐；不在 AC 里编造新的检查点。

### 验收报告

```
## 验收报告

### 通过（N/M）
- [x] AC1 — 证据：...
- [x] AC2 — 证据：...

### 未通过（如有）
- [ ] AC3 — 预期：X，实际：Y，修复方案：...

### 功能需求覆盖
- F1 → AC1, AC2：✅
- F2 → AC3：❌（见上）
```

## 危险信号

出现以下想法时，停下来——你在为跳过流程找理由：

| 想法                                | 现实                                                       |
| --------------------------------- | -------------------------------------------------------- |
| 「这个太简单了，不需要写 spec」                | 越简单的项目，未被审视的假设越多                                         |
| 「我直接写代码就行」                        | HARD GATE：spec 和 plan 全过了才能动代码                            |
| 「spec 太明显了，直接跳到 plan」             | 「明显」意味着没被验证过，写出来让用户确认                                    |
| 「验收标准等做完了再补」                      | AC 决定了你要交付什么，必须在编码前设计                                   |
| 「测试过了就说明没问题」                      | 测试验证代码，AC 验证需求                                          |
| 「应该没问题了」                          | 跑一下。先有证据再下结论                                             |
| 「我知道用户想要什么」                       | 问一下。一次一个问题                                               |
| 「自检通过了就够了」                        | 自检只是基线；必须经过 Explore agent 审查，通过率不是质量的证明                  |
| 「review 没找到问题，肯定写得很好」             | review 通过 ≠ 正确；**spec 用户审批**才是最后一道关                      |
| 「plan 不用用户审批，太省事了」                 | spec 必须人工审批（业务源头）；plan 软提示是设计选择，不是省事                    |

## 核心原则

- **一次一个问题**——每次聚焦一个问题
- **优先用选择题**——比开放题更容易回答
- **逐段审批**——逐步呈现，每段确认后再继续
- **层层递进**——每份文档在前一份基础上细化，逐步增加细节
- **YAGNI 铁律**——只设计和实现 spec 提到的内容
- **每个小节写完整**——所有内容就绪后再提交审批
- **先有证据再下结论**——先跑验证，再报告结果
- **审查通过 ≠ 正确**——Explore agent 报告是质量信号，但 **spec 用户审批**仍是最后一道关
- **审批强度分层**——spec 强审批（业务源头）、plan 软提示（默认推进）
- **审查策略由用户选**——每份文档生成前用 AskUserQuestion 询问单轮/多轮，默认单轮（节省 token）
- **聚焦行为描述**——spec 描述系统做什么，验收阶段按 AC 检验
